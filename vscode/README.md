# VSCode config

Per-host VSCode settings, generated by merging shared layers. VSCode's
built-in Settings Sync isn't used — it has no per-machine layer, so
work-only settings would leak to home machines.

## Layout

```
common/settings.json       user-global, applies on every host
fragments/work.json        atlascode + axivion, layered in for work hosts
hosts/<hostname>/          one dir per machine (named to match `hostname -s`)
  manifest.txt             newline-separated list of fragments to apply
  overlay.json             host-specific overrides (optional)
  extensions.txt           extension IDs to install on this host
  mcp.json                 host-specific MCP servers (optional)
bin/install-vscode.sh      merge settings + install missing extensions
workspace-template/        drop into a project's repo for per-project settings
```

Merge precedence (later wins): `common` → fragments listed in `manifest.txt` → `overlay.json`.

## Install on a new machine

```sh
cd ~/dotfiles
vscode/bin/install-vscode.sh
```

The script auto-detects the host via `hostname -s` and looks up
`hosts/<hostname>/`. It writes `settings.json` (merged), symlinks
keybindings/snippets/mcp, then `code --install-extension`s anything in
`extensions.txt` that isn't already installed (so re-runs after settings
tweaks are fast).

Requires `jq`. The `code` CLI is found on PATH or, on macOS, inside
`/Applications/Visual Studio Code.app` automatically.

## Adding a new host

```sh
mkdir vscode/hosts/$(hostname -s)
echo "work" > vscode/hosts/$(hostname -s)/manifest.txt   # optional
touch vscode/hosts/$(hostname -s)/extensions.txt          # add IDs as needed
vscode/bin/install-vscode.sh
```

## Per-project settings

Settings that describe a project (cmake presets, C++ standard, pytest
enabled, etc.) belong in the project's own `.vscode/settings.json`,
not in any host's User scope. `workspace-template/.vscode/settings.json`
has the keys extracted from the original MBP User settings — drop into
the Stafl repo (or similar) as a starting point.
